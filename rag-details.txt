================================================================================
 1. OVERVIEW & PURPOSE
================================================================================

The RAG (Retrieval-Augmented Generation) system is the knowledge intelligence
layer of the AI support agent backend. It combines a vector-based knowledge
base with multi-provider LLM generation and channel-aware formatting to
deliver accurate, context-grounded responses across all support channels.

The system spans four development sprints:

  Sprint 3: RAG Knowledge Base
    — Article CRUD, text chunking, sentence-transformer embeddings,
      pgvector semantic search, hybrid keyword+semantic search

  Sprint 4: PDF Document Integration
    — PDF text extraction via PyMuPDF, automated ingestion into the
      knowledge base, bulk ingest, upload endpoint

  Sprint 5: AI Response Providers
    — Multi-provider LLM support (OpenAI GPT, Anthropic Claude, Google Gemini),
      channel-aware formatting (CHAT, EMAIL, WHATSAPP, VOICE, TICKET),
      tone presets, streaming, multi-channel preview

  Sprint 6: Voice Agents RAG Integration
    — Internal API for service-to-service communication, RAG bridge HTTP
      client for voice agents, multi-provider LLM/STT/TTS factory,
      RAG function tools on all voice agents

KEY GOALS:
  • Ground AI responses in factual, up-to-date knowledge base content
  • Support multiple LLM providers (OpenAI, Claude, Gemini) with zero-code switching
  • Format responses automatically for each output channel (chat, email, WhatsApp, voice, ticket)
  • Enable voice agents to access RAG context in real-time during conversations
  • Provide full PDF-to-knowledge-base pipeline for document ingestion
  • Support semantic search (vector similarity) and hybrid search (keyword + semantic)
  • Fully async architecture with Celery background indexing for large batches

DESIGN PHILOSOPHY:
  • Provider-Agnostic — All LLM providers implement a common BaseProvider
    interface. Switching providers requires only changing an env var.
  • No SDK Lock-in — All provider implementations use raw httpx HTTP calls,
    avoiding heavy SDK dependencies.
  • Channel-Aware — Every response is automatically formatted for its target
    channel before delivery. Voice responses strip markdown; emails add
    greeting/sign-off; WhatsApp uses mobile-friendly formatting.
  • Graceful Degradation — If the RAG knowledge base has no relevant content,
    the system clearly indicates this. If a provider is down, voice agents
    fall back to their built-in LLM knowledge.
  • Fully Auditable — Every generation request logs provider, model, latency,
    token usage, and RAG chunks used.

    response generation, provider management

  • Voice Agents — Access RAG via internal API using service-key auth
  • LLM Providers — OpenAI, Claude, Gemini via httpx REST calls

 4. FULL RAG PIPELINE FLOW DIAGRAM
================================================================================

The complete pipeline from user query to formatted response:

    ┌─────────────────────────────────────────────────────────────────────────┐
    │                   FULL RAG RESPONSE GENERATION PIPELINE                  │
    └─────────────────────────────────────────────────────────────────────────┘

    USER QUERY
        │
        ▼
    ┌─────────────────────────────────────────────────────────┐
    │  STEP 1: RAG CONTEXT RETRIEVAL  (retriever.py)          │
    │                                                         │
    │  1a. Embed the query using sentence-transformers        │
    │      (all-MiniLM-L6-v2, 384 dimensions)                │
    │                                                         │
    │  1b. Execute pgvector cosine distance search:           │
    │      SELECT chunks with their articles                  │
    │      WHERE article.status = PUBLISHED                   │
    │            AND chunk.status = INDEXED                   │
    │            AND chunk.embedding IS NOT NULL              │
    │      ORDER BY cosine_distance ASC                       │
    │      LIMIT top_k (default 5)                            │
    │                                                         │
    │  1c. Filter by min_similarity threshold (default 0.3)   │
    │                                                         │
    │  OUTPUT:                                                │
    │    chunks = [                                           │
    │      {article_title, chunk_content, similarity, id},    │
    │      ...                                                │
    │    ]                                                    │
    │    sources = [SourceReference, ...]                      │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌─────────────────────────────────────────────────────────┐
    │  STEP 2: SYSTEM PROMPT ASSEMBLY  (base.py)              │
    │                                                         │
    │  Build system prompt from components:                   │
    │    a. Base role: "You are an AI-powered customer        │
    │       support assistant."                               │
    │                                                         │
    │    b. Channel instructions (CHAT/EMAIL/WHATSAPP/        │
    │       VOICE/TICKET) — see Section 31                    │
    │                                                         │
    │    c. Tone instructions (professional/friendly/         │
    │       empathetic/concise/technical)                     │
    │                                                         │
    │    d. Language directive (if specified)                  │
    │                                                         │
    │    e. Customer/agent name personalization               │
    │                                                         │
    │    f. RAG context injection:                            │
    │       "## Knowledge base context"                       │
    │       "### Source 1: <title>\n<content>"                │
    │       "### Source 2: <title>\n<content>"                │
    │       "Base your answer on the knowledge base context   │
    │        above. If the context doesn't contain enough     │
    │        information, say so clearly."                    │
    │                                                         │
    │  OUTPUT: Full system prompt string                      │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌─────────────────────────────────────────────────────────┐
    │  STEP 3: MESSAGE ASSEMBLY  (base.py)                    │
    │                                                         │
    │  Build OpenAI-compatible messages array:                │
    │    [                                                    │
    │      {role: "system", content: <system_prompt>},        │
    │      {role: "user", content: "history msg 1"},          │
    │      {role: "assistant", content: "history msg 2"},     │
    │      ... (last 10 conversation turns)                   │
    │      {role: "user", content: <current_query>},          │
    │    ]                                                    │
    │                                                         │
    │  NOTE: For Claude, system is a separate parameter.      │
    │        For Gemini, roles are remapped (assistant→model). │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌─────────────────────────────────────────────────────────┐
    │  STEP 4: LLM PROVIDER CALL  (provider implementations) │
    │                                                         │
    │  Selected by AI_RESPONSE_PROVIDER env var or per-request│
    │  override. All use httpx — no SDK dependencies.         │
    │                                                         │
    │  ┌─ OpenAI ─────────────────────────────────────────┐   │
    │  │ POST https://api.openai.com/v1/chat/completions  │   │
    │  │ Authorization: Bearer <OPENAI_API_KEY>            │   │
    │  │ Default model: gpt-4o-mini                        │   │
    │  └──────────────────────────────────────────────────┘   │
    │                                                         │
    │  ┌─ Claude ─────────────────────────────────────────┐   │
    │  │ POST https://api.anthropic.com/v1/messages        │   │
    │  │ x-api-key: <ANTHROPIC_API_KEY>                    │   │
    │  │ Default model: claude-sonnet-4-20250514           │   │
    │  └──────────────────────────────────────────────────┘   │
    │                                                         │
    │  ┌─ Gemini ─────────────────────────────────────────┐   │
    │  │ POST googleapis.com/.../generateContent           │   │
    │  │ key=<GEMINI_API_KEY>                              │   │

    │  └──────────────────────────────────────────────────┘   │
    │                                                         │
    │  OUTPUT:                                                │
    │    raw_response = "Based on our documentation..."       │
    │    model = "gpt-4o-mini"                                │
    │    tokens_used = 342                                    │
    │    latency_ms = 1850                                    │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌─────────────────────────────────────────────────────────┐
    │  STEP 5: CHANNEL FORMATTING  (channel_formatter.py)     │
    │                                                         │
    │  ┌─ CHAT ──────────────────────────────────────────┐    │
    │  │ Trim excessive whitespace, keep light markdown   │    │
    │  └─────────────────────────────────────────────────┘    │
    │                                                         │
    │  ┌─ EMAIL ─────────────────────────────────────────┐    │
    │  │ Ensure greeting + sign-off, use customer name    │    │
    │  └─────────────────────────────────────────────────┘    │
    │                                                         │
    │  ┌─ WHATSAPP ──────────────────────────────────────┐    │
    │  │ Strip markdown, use • bullets, mobile-friendly   │    │
    │  └─────────────────────────────────────────────────┘    │
    │                                                         │
    │  ┌─ VOICE ─────────────────────────────────────────┐    │
    │  │ Strip ALL formatting, expand abbreviations,      │    │
    │  │ remove URLs, natural spoken language              │    │
    │  └─────────────────────────────────────────────────┘    │
    │                                                         │
    │  ┌─ TICKET ────────────────────────────────────────┐    │
    │  │ Keep full markdown, append source references     │    │
    │  └─────────────────────────────────────────────────┘    │
    │                                                         │
    │  OUTPUT: Formatted response string                      │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌─────────────────────────────────────────────────────────┐
    │  FINAL OUTPUT: GenerateResponse                         │
    │                                                         │
    │  {                                                      │
    │    response: "Formatted text for the channel",          │
    │    raw_response: "Unformatted LLM output",              │
    │    provider: "openai",                                  │
    │    model_used: "gpt-4o-mini",                           │
    │    channel: "VOICE",                                    │

 6. MODULE 2: TEXT CHUNKER (chunker.py)
================================================================================

PURPOSE:
  Splits long documents into overlapping text chunks suitable for embedding
  and retrieval. Uses a recursive strategy that respects natural text
  boundaries (paragraphs → sentences → words → characters).

CLASS: RecursiveTextChunker

CONFIGURATION (ChunkerConfig):
  ┌────────────────────────┬──────────┬──────────────────────────────────────┐
  │ Parameter              │ Default  │ Description                          │
  ├────────────────────────┼──────────┼──────────────────────────────────────┤
  │ chunk_size             │ 512      │ Target tokens per chunk              │
  │ chunk_overlap          │ 64       │ Overlap tokens between chunks        │
  │ separators             │ (list)   │ Split hierarchy (see below)          │
  │ strip_whitespace       │ True     │ Trim whitespace from chunk edges     │
  └────────────────────────┴──────────┴──────────────────────────────────────┘

SEPARATOR HIERARCHY (priority order):
  1. "\n\n"  — double newline (paragraph breaks) — preferred split point
  2. "\n"    — single newline
  3. ". "    — sentence boundary
  4. "? "    — question mark
  5. "! "    — exclamation mark
  6. "; "    — semicolon
  7. ", "    — comma
  8. " "     — word boundary (last resort before force-split)

ALGORITHM:

    ┌─────────────────────────────────────────────────────────┐
    │            RECURSIVE TEXT CHUNKING ALGORITHM             │
    └─────────────────────────────────────────────────────────┘

    Input: "Long document text..." (e.g., 5000 words)
           │
           ▼
    ┌── STEP 1: _split_recursive(text, sep_idx=0) ───────────┐
    │                                                         │
    │  Is text small enough? (≤ chunk_size tokens)            │
    │    YES → Return [text]                                  │
    │    NO  → Split by separator[0] = "\n\n" (paragraphs)   │
    │                                                         │
    │  For each resulting piece:                              │
    │    Small enough? → Keep                                 │
    │    Too large?    → Recurse with separator[1] = "\n"     │
    │      Still too large? → separator[2] = ". "             │
    │      Still too large? → separator[3] = "? "             │
    │      ... continue down the hierarchy ...                │
    │      All separators exhausted? → _force_split()         │
    │        (split by character count as last resort)         │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌── STEP 2: _merge_small_chunks(pieces) ──────────────────┐
    │                                                         │
    │  Merge consecutive small pieces until reaching target:  │
    │                                                         │
    │  current = ""                                           │
    │  for piece in pieces:                                   │
    │    candidate = current + " " + piece                    │
    │    if tokens(candidate) <= chunk_size:                   │
    │      current = candidate    ← merge into current        │
    │    else:                                                │
    │      output current, reset current = piece              │
    │                                                         │
    │  This prevents tiny orphan chunks.                      │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌── STEP 3: _add_overlap(chunks) ─────────────────────────┐
    │                                                         │
    │  For each chunk[i] (except first):                      │
    │    Take the last N words from chunk[i-1]                │
    │    Prepend them to chunk[i]                             │
    │                                                         │
    │  N = chunk_overlap (default 64 words)                   │
    │                                                         │
    │  This creates semantic continuity between chunks,       │
    │  ensuring that information at chunk boundaries is        │
    │  captured in both adjacent chunks.                       │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    Output: List of Chunk(index, content, token_count)

TOKEN ESTIMATION:
  estimate_tokens(text) → count whitespace-separated tokens
  Approximately 1 token per word (rough estimate for English)

CONVENIENCE FUNCTION:
  chunk_text(text, chunk_size=512, chunk_overlap=64) → list[Chunk]
  Creates a chunker with defaults and returns chunks.

  │ Method                   │ Description                                   │
  ├──────────────────────────┼───────────────────────────────────────────────┤
  │ semantic_search(request) │ Pure vector similarity search using pgvector  │
  │                          │ cosine distance operator.                     │
  ├──────────────────────────┼───────────────────────────────────────────────┤
  │ hybrid_search(request)   │ Combined keyword + semantic search with       │
  │                          │ configurable weighting.                       │
  ├──────────────────────────┼───────────────────────────────────────────────┤
  │ get_context_for_query()  │ Simplified helper that returns a list of      │
  │                          │ dicts for internal use (decision engine,      │
  │                          │ response providers).                          │
  └──────────────────────────┴───────────────────────────────────────────────┘

SEMANTIC SEARCH ALGORITHM:

    Query: "How do I reset my password?"
           │
           ▼
    1. embed_text(query) → [0.032, -0.118, ..., 0.074]  (384 floats)
           │
           ▼
    2. SQL: SELECT chunk.*, article.title, article.category,
              (1 - chunk.embedding <=> query_embedding) AS similarity
            FROM article_chunks chunk
            JOIN knowledge_articles article ON chunk.article_id = article.id
            WHERE article.status = 'PUBLISHED'
              AND article.is_deleted = FALSE
              AND chunk.status = 'INDEXED'
              AND chunk.embedding IS NOT NULL
              [AND article.category = <filter>]   -- optional
              [AND article.language = <filter>]   -- optional
            ORDER BY chunk.embedding <=> query_embedding ASC
            LIMIT <top_k>
           │
           ▼
    3. Filter: Remove hits where similarity < min_similarity
           │
           ▼
    4. Return SearchResponse(query, hits, total_hits, model_used)

    NOTE: <=> is pgvector's cosine distance operator.
          cosine_distance = 1 - cosine_similarity
          Lower distance = higher similarity.

  See Section 29 for the detailed hybrid scoring formula.

  • Uses the same OpenAI-compatible /v1/chat/completions format that all

  4. Yield text tokens

  1. GET <LOCAL_LLM_BASE_URL>/models → 200 = healthy

================================================================================
 29. EMBEDDING & SIMILARITY ALGORITHMS (DETAILED)
================================================================================

═══ A. EMBEDDING GENERATION ═══

  Text → sentence-transformers → normalized vector ∈ ℝ³⁸⁴

  Model: all-MiniLM-L6-v2
    • Based on MiniLM architecture (distilled from BERT)
    • 22 million parameters
    • 384-dimensional output
    • Trained on 1B+ sentence pairs
    • L2-normalization applied (normalize_embeddings=True)

  Process:
    1. Tokenize input text using model's tokenizer
    2. Pass through 6-layer transformer
    3. Mean pooling over token embeddings
    4. L2-normalize the result → unit vector on 384-dim hypersphere

  Since all vectors are L2-normalized:
    cosine_similarity(a, b) = dot_product(a, b)
    cosine_distance(a, b)   = 1 - dot_product(a, b)

═══ B. COSINE SIMILARITY SEARCH ═══

  Given query embedding q and chunk embeddings {c₁, c₂, ..., cₙ}:

    similarity(q, cᵢ) = q · cᵢ / (‖q‖ × ‖cᵢ‖)

  Since vectors are normalized (‖q‖ = ‖cᵢ‖ = 1):

    similarity(q, cᵢ) = q · cᵢ

  PostgreSQL query uses the <=> operator (cosine distance):

    distance = 1 - similarity
    ORDER BY distance ASC = ORDER BY similarity DESC

  Results are filtered: similarity ≥ min_similarity (default 0.3)

═══ C. BATCH EMBEDDING ═══

  For indexing, chunks are embedded in batches:
    • Default batch_size: 64
    • Progress bar shown for >100 texts
    • Returns list of 384-dim vectors

  Example: 10 chunks → embed_texts(10 texts, batch=64)
    → 1 batch → 10 × 384 matrix → 10 normalized vectors


================================================================================
 30. CHUNKING ALGORITHM (DETAILED)
================================================================================

═══ RECURSIVE SPLITTING ═══

  The RecursiveTextChunker uses a separator hierarchy to split text at
  natural boundaries while respecting the target chunk size.

  Input: "Long document..." (2000 words)
  Config: chunk_size=512, chunk_overlap=64

  PHASE 1: Split by separators

    Try "\n\n" (paragraphs):
      "Paragraph 1" (100 words) → OK, keep
      "Paragraph 2" (800 words) → TOO LARGE, recurse
        Try "\n" (newlines):
          "Section A" (300 words) → OK, keep
          "Section B" (500 words) → OK, keep
      "Paragraph 3" (200 words) → OK, keep
      ...

  PHASE 2: Merge small chunks

    ["100-word piece", "50-word piece", "80-word piece", ...]
    → Merge consecutive small pieces until hitting chunk_size:
      "100 + 50 + 80 = 230 words" → still < 512, keep merging
      "230 + 300 = 530 words" → exceeds 512, split here

  PHASE 3: Add overlap

    Chunk[0]: "...end of chunk 0"
    Chunk[1]: "end of chunk 0 [OVERLAP] start of chunk 1..."
    Chunk[2]: "end of chunk 1 [OVERLAP] start of chunk 2..."

    Overlap = last 64 words of previous chunk prepended to next

═══ TOKEN ESTIMATION ═══

    estimate_tokens(text)  = count of whitespace-separated tokens
    Approximation:           ~1 token per word (for English)
    Used for:                chunk size decisions (not billing)

═══ FORCE SPLIT ═══

  If all separators are exhausted and text is still too long:
    Split by character count: chunk_size × 4 chars per chunk
    This is a last resort for texts with no natural boundaries.


================================================================================
 31. HYBRID SEARCH SCORING ALGORITHM
================================================================================

The hybrid search combines keyword matching with semantic similarity
using a weighted formula.

═══ FORMULA ═══

    hybrid_score = keyword_weight × kw_score + (1 - keyword_weight) × semantic_score

  Where:
    keyword_weight = request.keyword_weight (default 0.3)
    kw_score = number_of_matching_keywords / total_keywords  (0.0 to 1.0)
    semantic_score = 1 - cosine_distance  (0.0 to 1.0)

═══ ALGORITHM ═══

    Input: query = "SMS API integration", keyword_weight = 0.3
           │
           ▼
    1. Embed query → query_embedding (384 dims)
           │
           ▼
    2. Extract keywords: ["sms", "api", "integration"]
           │
           ▼
    3. Execute pgvector search (fetch top_k × 3 candidates):
       SELECT chunks ordered by cosine distance ASC
           │
           ▼
    4. For each candidate chunk:
       a. semantic_score = 1 - cosine_distance (from pgvector)
       b. Count keyword matches in chunk.content.lower():
          "sms" found? YES (+1)
          "api" found? YES (+1)
          "integration" found? NO (+0)
          kw_score = 2/3 = 0.667
       c. hybrid_score = 0.3 × 0.667 + 0.7 × 0.82 = 0.200 + 0.574 = 0.774
           │
           ▼
    5. Filter: hybrid_score ≥ min_similarity (default 0.2)
           │
           ▼
    6. Sort by hybrid_score DESC, take top_k
           │
           ▼
    Output: SearchResponse with re-ranked hits

═══ EXAMPLE ═══

    Chunk A: "Our SMS API supports REST..." 
      semantic = 0.82, keywords = 3/3 = 1.0
      hybrid = 0.3×1.0 + 0.7×0.82 = 0.874

    Chunk B: "Message delivery notification..."
      semantic = 0.85, keywords = 0/3 = 0.0
      hybrid = 0.3×0.0 + 0.7×0.85 = 0.595

    Chunk A ranks higher despite lower semantic score because
    all keywords match. The keyword component provides a lexical
    signal that complements the semantic embedding.

 33. CHANNEL FORMATTING RULES
================================================================================

═══ CHAT ═══
  Input:  "Here is the info:\n\n\n\nPoint 1: ..."
  Output: "Here is the info:\n\nPoint 1: ..."
  Rules:  Collapse ≥3 consecutive newlines to 2. Light markdown OK.

═══ EMAIL ═══
  Input:  "Your account has been updated."
  Output: "Hello John,\n\nYour account has been updated.\n\nBest regards,\nSupport Team"
  Rules:
    • Detect missing greeting → add "Hello <customer_name>,"
    • Detect missing sign-off → add "Best regards,\n<agent_name>"
    • Greeting detection regex: /^(dear|hello|hi|hey|good morning|...)/i
    • Sign-off detection regex: /(best regards|kind regards|sincerely|...)\s*[,.]?\s*$/im

═══ WHATSAPP ═══
  Input:  "## Header\n**Bold text**\n- Item 1\n- Item 2\n[Link](http://...)"
  Output: "Header\n*Bold text*\n• Item 1\n• Item 2\nLink (http://...)"
  Rules:
    • Remove # headers → plain text
    • **bold** → *bold* (WhatsApp native bold)
    • Remove image markdown
    • [text](url) → text (url)
    • "- " bullets → "• " bullets
    • Collapse blank lines

═══ VOICE ═══
  Input:  "## Steps\n1. Go to **Settings**\n2. Click `Reset`\nhttps://example.com"
  Output: "Steps. Go to Settings. Click Reset."
  Rules:
    • Strip ALL markdown: headers, bold, italic, code
    • Remove ALL URLs
    • Remove bullet and numbered list markers
    • Multi-newlines → ". " (sentence break)
    • Collapse whitespace
    • Expand abbreviations:
      "e.g." → "for example"
      "i.e." → "that is"
      "etc." → "and so on"
      "vs."  → "versus"
      "approx." → "approximately"

═══ TICKET ═══
  Input:  "Here is detailed analysis..."
  Output: "Here is detailed analysis...\n\n---\n**References:**\n1. Title (relevance: 85%)\n2. ..."
  Rules:
    • Keep all markdown formatting intact
    • Append source references section (if sources provided)
    • Only append if "source" not already in the last 200 chars

═══ RAG FUNCTION TOOLS ON ALL AGENTS ═══

  All voice agents inherit from GenericAgent, which provides two RAG tools:

  @function_tool
  search_knowledge_base(query: str) → str
    "Search the knowledge base for information about TunisieSMS services..."
    • Calls rag_bridge.search_knowledge_base(query, top_k=5)
    • Returns formatted context or "No relevant information found..."
    • Used when the LLM decides to look up information

  @function_tool
  generate_answer(question: str) → str
    "Generate a detailed answer using the RAG knowledge base and AI providers."
    • Calls rag_bridge.generate_rag_response(question, channel="VOICE", tone="friendly")
    • Returns the generated response or fallback message
    • Used for comprehensive, well-sourced answers

  The LLM decides when to call these tools based on the conversation context.
  Tool descriptions guide the LLM on when each tool is appropriate.

═══ MULTI-PROVIDER VOICE PIPELINE ═══

  The voice pipeline components (STT, LLM, TTS) are selected by factory
  functions in llm_factory.py using a priority chain.  Local / free
  engines are the DEFAULT; cloud providers activate only when their
  credentials are present.

  ── STT priority (make_stt) ──
    1. openai.STT()            — if OPENAI_API_KEY set ($$$)
    2. google.STT()            — if GOOGLE_APPLICATION_CREDENTIALS set ($$$)
    3. LocalWhisperSTT()       — DEFAULT / FREE (faster-whisper, CPU, int8)

  ── LLM priority (make_llm) ──
    • USE_REALTIME=true  → google.beta.realtime.RealtimeModel(voice)
    • provider=openai    → openai.LLM(model="gpt-4o-mini")
    • default (gemini)   → google.LLM(model="gemini-2.5-flash-lite")
      (works with free Gemini API key via GOOGLE_API_KEY)

  ── TTS priority (make_tts) ──
    1. None               — if USE_REALTIME (realtime model handles TTS)
    2. openai.TTS()       — if provider=openai AND OPENAI_API_KEY ($$$)
    3. google.TTS()       — if GOOGLE_APPLICATION_CREDENTIALS set ($$$)
    4. EdgeTTS()          — DEFAULT / FREE (edge-tts, Microsoft neural voices)

  ── Voice-name mapping (EdgeTTS) ──
    Gemini voice   →  Edge TTS voice
    ─────────────────────────────────
    Puck           →  en-US-GuyNeural
    Charon         →  en-US-ChristopherNeural
    Aoede          →  en-US-JennyNeural
    Kore           →  en-US-AriaNeural
    (other)        →  en-GB-RyanNeural (default)

         │

         │
         ▼
    ┌──────────┐
    │  DRAFT   │  ← Indexed but still not searchable
    │ (indexed) │
    └────┬─────┘
         │ POST /rag/articles/{id}/publish
         ▼
    ┌──────────────┐
    │  PUBLISHED   │  ← Now searchable by the retriever
    └────┬─────────┘

         ▼
    ┌──────────────┐
    │  ARCHIVED    │  ← Removed from search results
    └──────────────┘
         │ DELETE /rag/articles/{id}
         ▼
    ┌──────────────┐
    │  SOFT-DELETED│  ← is_deleted=true, invisible
    └──────────────┘


═══ DIAGRAM B: PDF Ingestion Pipeline ═══

    PDF File (e.g., "getting-started-guide.pdf")
         │
         ▼
    ┌─ extract_pdf() ─────────────────────────────────────────┐
    │  fitz.open(filepath)                                     │
    │  → Extract metadata (title, author, subject)             │
    │  → For each page: get_text("text")                       │
    │  → Count chars, words per page                           │
    │  → Return PDFDocument(pages, metadata, totals)           │
    └────────────────────────┬────────────────────────────────┘
                             │
                             ▼
    ┌─ ingest_pdf() ──────────────────────────────────────────┐
    │  1. Create ArticleCreate payload:                        │
    │     title = PDF title or filename-derived                │
    │     content = pdf_doc.text_only                          │
    │     source = "pdf:getting-started-guide.pdf"             │
    │     tags = ["pdf", "getting-started-guide.pdf"]          │
    │     metadata_extra = {pdf_pages, pdf_words, ...}         │
    │                                                          │
    │  2. kb_service.create_article() → DRAFT                  │
    │                                                          │
    │  3. If auto_index: kb_service.index_article()            │
    │     → Chunk (RecursiveTextChunker)                       │
    │     → Embed (sentence-transformers)                      │
    │     → Store chunks with vector(384) embeddings           │
    │                                                          │
    │  4. If auto_publish: kb_service.publish_article()        │
    │     → status = PUBLISHED → now searchable                │
    └──────────────────────────────────────────────────────────┘
