 1. OVERVIEW & PURPOSE
================================================================================

The AI Decision Engine is the intelligent core of the support ticket management
system. It automatically analyzes incoming customer support tickets and makes
decisions about how each ticket should be handled — whether it can be
auto-resolved, needs a suggested response for an agent, requires clarification
from the customer, should be escalated to a human specialist, or should be
routed to a specific skilled agent.

KEY GOALS:
  • Reduce agent workload by auto-resolving simple, high-confidence tickets
  • Accelerate response times with AI-generated response suggestions
  • Ensure high-risk or security-sensitive tickets are escalated immediately
  • Route tickets to the best-suited agent based on skills and current workload
  • Provide full traceability of every AI decision for audit and improvement
  • Support bilingual detection (French & English keywords)

DESIGN PHILOSOPHY:
  • Deterministic & Interpretable — No black-box ML models. Every decision 
    can be traced back to matched keywords, risk factors, and applied rules.
  • Rule-Based with Keyword Matching — Uses weighted keyword dictionaries and
    regex patterns instead of LLMs for classification. LLMs are reserved for
    future response reformulation only.
  • Configurable — All rules, keywords, thresholds, and weights are defined as
    data structures that can be modified without code changes.
  • Fully Auditable — Every decision is persisted to the database with full
    reasoning, matched rules, and scores.

 5. MODULE 1: INTENT CLASSIFIER (classifier.py)
================================================================================

PURPOSE:
  Determines what a customer's support ticket is about by matching the text
  against predefined keyword dictionaries and regex patterns for 8 categories.

ALGORITHM:

  1. COMBINE text: subject + description → one string
  2. NORMALIZE: lowercase, collapse whitespace
  3. For EACH category rule:
     a. Match keywords: +1.0 per keyword hit (multi-word keywords get +0.5 bonus)
     b. Match regex patterns: +2.0 per pattern hit (patterns are more specific)
     c. Apply category weight multiplier
  4. RANK categories by score (descending)
  5. COMPUTE confidence:
     • dominance_ratio = best_score / total_score (how dominant is the winner)
     • absolute_factor = min(best_score / 10, 1.0) (absolute strength)
     • confidence = 0.6 × dominance_ratio + 0.4 × absolute_factor
  6. MAP confidence to level:
     • >= 0.70 → HIGH
     • >= 0.40 → MEDIUM
     • < 0.40  → LOW

CATEGORY KEYWORD COUNTS:
  ┌──────────────────┬──────────┬──────────┬────────┐
  │ Category         │ Keywords │ Patterns │ Weight │
  ├──────────────────┼──────────┼──────────┼────────┤
  │ BILLING          │ 25       │ 4        │ 1.0    │
  │ TECHNICAL        │ 30       │ 3        │ 1.0    │
  │ ACCOUNT          │ 22       │ 2        │ 1.0    │
  │ COMPLAINT        │ 19       │ 2        │ 1.2    │
  │ FEATURE_REQUEST  │ 16       │ 2        │ 0.8    │
  │ SECURITY         │ 18       │ 2        │ 1.3    │
  │ URGENT           │ 15       │ 2        │ 1.5    │
  │ GENERAL          │ (none)   │ (none)   │ —      │
  └──────────────────┴──────────┴──────────┴────────┘

  NOTE: GENERAL is the fallback when no keywords match. It is not a scored
  category — it triggers only when all other categories score zero.

WEIGHT RATIONALE:
  • URGENT (1.5×) — Highest weight because urgency affects all categories.
    Even a few urgency keywords should override other signals.
  • SECURITY (1.3×) — Security is high-stakes; even minimal security keywords
    should strongly influence classification.
  • COMPLAINT (1.2×) — Complaints can escalate quickly; boosted slightly.
  • FEATURE_REQUEST (0.8×) — Lower weight because feature request keywords 
    are sometimes used colloquially. Reduces false positives.
  • All others (1.0×) — Standard weight.

BILINGUAL SUPPORT:
  Each category includes French keywords alongside English. Examples:
    BILLING:    "facture", "paiement", "remboursement", "abonnement", "tarif"
    TECHNICAL:  "erreur", "problème", "ne fonctionne pas", "lent", "bloqué"
    ACCOUNT:    "compte", "mot de passe", "connexion", "inscription"
    COMPLAINT:  "plainte", "mécontent", "insatisfait", "déçu", "inacceptable"
    SECURITY:   "sécurité", "piratage", "piraté", "suspect", "vol"
    URGENT:     "urgente", "critique", "immédiatement"

CONFIDENCE CALCULATION DIAGRAM:

    Example: Text matches BILLING (score=5.0) and TECHNICAL (score=2.0)

    Total Score    = 5.0 + 2.0 = 7.0
    Dominance      = 5.0 / 7.0 = 0.714
    Absolute       = min(5.0 / 10.0, 1.0) = 0.5
    Confidence     = 0.6 × 0.714 + 0.4 × 0.5 = 0.429 + 0.200 = 0.629

    Result: BILLING with 62.9% confidence → MEDIUM level

 6. MODULE 2: RISK & CONFIDENCE SCORER (scorer.py)
================================================================================

PURPOSE:
  Computes a risk score (0.0–1.0) by analyzing the text for negative sentiment,
  urgency markers, escalation language, security indicators, and contextual
  factors like existing ticket priority and escalation history.

NINE RISK FACTORS:

  ┌────┬──────────────────────────────┬────────────────┬──────────────────────┐
  │ #  │ Factor                       │ Max Impact     │ How It Works         │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 1  │ Category base risk           │ 0.0 – 0.21    │ Each category has an │
  │    │                              │ (0.7 × 0.3)   │ inherent risk level  │
  │    │                              │                │ (SECURITY=0.7,       │
  │    │                              │                │  URGENT=0.6, etc.)   │
  │    │                              │                │ Multiplied by 0.3    │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 2  │ Negative sentiment           │ 0.0 – 0.20    │ Count of negative    │
  │    │                              │                │ words × 0.05, max    │
  │    │                              │                │ 0.20. Words like:    │
  │    │                              │                │ angry, furious, hate,│
  │    │                              │                │ scam, lawsuit, etc.  │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 3  │ Urgency markers              │ 0.0 – 0.15    │ Count × 0.05,        │
  │    │                              │                │ max 0.15. Words:     │
  │    │                              │                │ urgent, ASAP,        │
  │    │                              │                │ emergency, critical  │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 4  │ Escalation triggers          │ 0.0 – 0.15    │ Count × 0.05,        │
  │    │                              │                │ max 0.15. Words:     │
  │    │                              │                │ manager, supervisor, │
  │    │                              │                │ escalate, not        │
  │    │                              │                │ resolved             │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 5  │ Security risk words          │ 0.0 – 0.15    │ Count × 0.05,        │
  │    │                              │                │ max 0.15. Words:     │
  │    │                              │                │ breach, hacked,      │
  │    │                              │                │ unauthorized, data   │
  │    │                              │                │ leak                 │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 6  │ Text length extremes         │ 0.0 – 0.05    │ <5 words: +0.05      │
  │    │                              │                │ >500 words: +0.03    │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 7  │ Existing priority level      │ -0.03 – +0.10 │ CRITICAL: +0.10      │
  │    │                              │                │ HIGH: +0.05          │
  │    │                              │                │ MEDIUM: 0.0          │
  │    │                              │                │ LOW: -0.03           │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 8  │ Escalation flag              │ +0.10          │ If ticket was        │
  │    │                              │                │ previously flagged   │
  │    │                              │                │ for escalation       │
  ├────┼──────────────────────────────┼────────────────┼──────────────────────┤
  │ 9  │ Low confidence amplifier     │ 0.0 – 0.08    │ LOW conf: +0.08      │
  │    │                              │                │ MEDIUM conf: +0.03   │
  │    │                              │                │ HIGH conf: +0.0      │
  └────┴──────────────────────────────┴────────────────┴──────────────────────┘

  THEORETICAL MAX RISK = 0.21 + 0.20 + 0.15 + 0.15 + 0.15 + 0.05 + 0.10 
                         + 0.10 + 0.08 = 1.19 → clamped to 1.0

CATEGORY BASE RISK VALUES:
  ┌──────────────────┬────────────────┐
  │ Category         │ Base Risk      │
  ├──────────────────┼────────────────┤
  │ SECURITY         │ 0.70           │
  │ URGENT           │ 0.60           │
  │ COMPLAINT        │ 0.50           │
  │ BILLING          │ 0.30           │
  │ TECHNICAL        │ 0.25           │
  │ ACCOUNT          │ 0.20           │
  │ FEATURE_REQUEST  │ 0.10           │
  │ GENERAL          │ 0.10           │
  └──────────────────┴────────────────┘
  Note: Base risk is multiplied by 0.3, so max category contribution is 0.21.

RISK LEVEL THRESHOLDS:
  ┌────────────────┬───────────────┬──────────────────────┐
  │ Risk Level     │ Score Range   │ Suggested Priority   │
  ├────────────────┼───────────────┼──────────────────────┤
  │ CRITICAL       │ >= 0.70       │ CRITICAL             │
  │ HIGH           │ >= 0.50       │ HIGH                 │
  │ MEDIUM         │ >= 0.30       │ MEDIUM               │
  │ LOW            │ < 0.30        │ LOW                  │
  └────────────────┴───────────────┴──────────────────────┘

 7. MODULE 3: RULE-BASED DECISION LOGIC (rules.py)
================================================================================

PURPOSE:
  Maps the combination of (confidence_level, risk_level, category) to a
  DecisionOutcome using a priority-ordered rule set. The first matching rule
  wins — earlier rules have higher priority.

RULE EVALUATION ORDER (highest priority first):

  ┌─────┬────────────────────────────────────────┬────────────┬────────────┬───────────────────┐
  │ Pri │ Rule Name                              │ Confidence │ Risk       │ Outcome           │
  ├─────┼────────────────────────────────────────┼────────────┼────────────┼───────────────────┤
  │ 100 │ critical_risk_always_escalate          │ ANY        │ CRITICAL   │ ESCALATE_HUMAN    │
  │  95 │ security_always_escalate (SECURITY)    │ ANY        │ ANY        │ ESCALATE_HUMAN    │
  │  90 │ low_confidence_escalate                │ LOW        │ ANY        │ ESCALATE_HUMAN    │
  │  85 │ high_risk_escalate                     │ HIGH/MED   │ HIGH       │ ESCALATE_HUMAN    │
  │  70 │ high_confidence_low_risk_auto_resolve  │ HIGH       │ LOW        │ AUTO_RESOLVE      │
  │  60 │ high_confidence_medium_risk_suggest    │ HIGH       │ MEDIUM     │ SUGGEST_RESPONSE  │
  │  50 │ medium_confidence_low_risk_suggest     │ MEDIUM     │ LOW        │ SUGGEST_RESPONSE  │
  │  40 │ medium_confidence_medium_risk_clarify  │ MEDIUM     │ MEDIUM     │ CLARIFY           │
  │  — │ fallback_route_agent                   │ (no match) │ (no match) │ ROUTE_AGENT       │
  └─────┴────────────────────────────────────────┴────────────┴────────────┴───────────────────┘

SPECIAL CATEGORY RULE:
  • The "security_always_escalate" rule has category=SECURITY constraint.
    It fires ONLY for SECURITY tickets, but at ALL confidence and risk levels.
    This means: any ticket classified as SECURITY is ALWAYS escalated to a 
    human, regardless of confidence or risk score.

RULE EVALUATION FLOW:

    Input: confidence=HIGH, risk=LOW, category=ACCOUNT
           │
           ▼
    ┌─ Rule: critical_risk_always_escalate ──┐
    │  risk=CRITICAL? NO → skip              │
    └────────────────────────────────────────┘
           │
           ▼
    ┌─ Rule: security_always_escalate ───────┐
    │  category=SECURITY? NO → skip          │
    └────────────────────────────────────────┘
           │
           ▼
    ┌─ Rule: low_confidence_escalate ────────┐
    │  confidence=LOW? NO → skip             │
    └────────────────────────────────────────┘
           │
           ▼
    ┌─ Rule: high_risk_escalate ─────────────┐
    │  risk=HIGH? NO → skip                  │
    └────────────────────────────────────────┘
           │
           ▼
    ┌─ Rule: high_confidence_low_risk ───────┐
    │  confidence=HIGH? YES                  │
    │  risk=LOW? YES                         │
    │  → MATCH! outcome = AUTO_RESOLVE       │  ← FIRST MATCH WINS
    └────────────────────────────────────────┘

 8. MODULE 4: DECISION ENGINE ORCHESTRATOR (decision_engine.py)
================================================================================

PURPOSE:
  The central orchestrator that chains all modules together into a single
  coherent pipeline. It coordinates the 9-step process detailed in Section 4.

TWO ENTRY POINTS:

  1. analyze_ticket(db, ticket, auto_assign, auto_update_priority)
     → Full pipeline with DB persistence and auto-actions
     → Requires a Ticket ORM object and DB session
     → Persists DecisionLog to database
     → Can auto-assign agent and auto-update priority

  2. analyze_text_only(text, subject)
     → Lightweight preview pipeline — NO database operations
     → Runs steps 1-4 only (classify, score, decide, suggest)
     → Uses a zero UUID as placeholder ticket_id
     → Useful for previewing classification before ticket creation

REASONING BUILDER FORMAT:
  The _build_reasoning() function generates a human-readable explanation:

  "Intent classified as ACCOUNT with 70.9% confidence (HIGH). Matched keywords:
  login, password, reset, locked out, locked. Risk assessed at 16.0% (LOW).
  Risk factors: Urgency detected (1 markers). Decision: AUTO_RESOLVE (rules:
  high_confidence_low_risk_auto_resolve)."

AUTO-ACTIONS (Step 9):
  These are optional side effects that modify the ticket:

  ┌───────────────────────┬───────────────────────────────────────────────┐
  │ Action                │ What Happens                                  │
  ├───────────────────────┼───────────────────────────────────────────────┤
  │ auto_update_priority  │ If risk.suggested_priority ≠ ticket.priority, │
  │                       │ update the ticket's priority in the DB        │
  ├───────────────────────┼───────────────────────────────────────────────┤
  │ auto_assign           │ If a suggested_agent was found and the ticket │
  │                       │ isn't already assigned, assign the agent and  │
  │                       │ set status to IN_PROGRESS                     │
  ├───────────────────────┼───────────────────────────────────────────────┤
  │ auto_escalate         │ If outcome=ESCALATE_HUMAN, set ticket status  │
  │ (always on)           │ to ESCALATED and escalation_flag=True         │
  └───────────────────────┴───────────────────────────────────────────────┘

 9. MODULE 5: SMART ROUTING ENGINE (router_engine.py)
================================================================================

PURPOSE:
  Finds the best-suited agent to handle a ticket based on the classified intent
  category, agent skill proficiency, and current workload.

ROUTING ALGORITHM:

    ┌───────────────────────────────────────────────────────────┐
    │              SMART ROUTING ALGORITHM                      │
    └───────────────────────────────────────────────────────────┘

    Start: category = ACCOUNT, ticket_id = abc-123
           │
           ▼
    ┌── STEP 1: Skill-Based Matching ────────────────────────┐
    │                                                         │
    │  Query: Find all AgentSkill records where:              │
    │    • skill_category = ACCOUNT                           │
    │    • agent.role IN (AGENT, ADMIN)                       │
    │    • agent.status = ACTIVE                              │
    │    • agent.is_deleted = False                           │
    │                                                         │
    │  For each matched agent:                                │
    │    • Count open/in-progress tickets (workload)          │
    │    • Skip if workload >= max_concurrent_tickets         │
    │    • Score = proficiency × (1 - workload_ratio)         │
    │                                                         │
    │  Example:                                               │
    │    Agent A: proficiency=0.85, workload=3/15             │
    │      score = 0.85 × (1 - 3/15) = 0.85 × 0.8 = 0.68   │
    │    Agent B: proficiency=0.70, workload=1/10             │
    │      score = 0.70 × (1 - 1/10) = 0.70 × 0.9 = 0.63   │
    │                                                         │
    │  Sort by score (descending) → Select Agent A            │
    └────────────────────┬────────────────────────────────────┘
                         │
              ┌──────────┴──────────┐
              │ Skill match found?  │
              ├─── YES ─────────────┼─── NO ────────────┐
              │                     │                    │
              ▼                     │                    ▼
    ┌── Return best match ──┐      │    ┌── STEP 2: Fallback ──────────────┐
    │  with all candidates  │      │    │                                   │
    │  ranked by score      │      │    │  Query: ALL active agents         │
    └───────────────────────┘      │    │  (AGENT or ADMIN role)            │
                                   │    │                                   │
                                   │    │  For each: count current workload │
                                   │    │  Score = 0.0 (no skill match)     │
                                   │    │  Sort by workload ASC             │
                                   │    │  → Select LEAST loaded agent      │
                                   │    │                                   │
                                   │    │  Reasoning: "Fallback routing     │
                                   │    │  (no skill match for ACCOUNT)"    │
                                   │    └───────────────────────────────────┘

SCORING FORMULA:

    Score = Proficiency × (1 - Current_Workload / Max_Concurrent_Tickets)

    Where:
      • Proficiency: 0.0–1.0 (from AgentSkill record)
      • Current_Workload: count of OPEN + IN_PROGRESS tickets assigned
      • Max_Concurrent_Tickets: from AgentSkill record (default 10)

    This ensures:
      • Higher-skilled agents are preferred
      • Agents with more available capacity are preferred
      • An agent at full capacity (workload = max) gets score = 0
      • An idle skilled agent (workload = 0) gets score = proficiency

 13. MODULE 9: CELERY ASYNC TASKS (tasks.py)
================================================================================

PURPOSE:
  Provides fire-and-forget asynchronous ticket analysis via Celery workers.
  Allows the API to immediately return a response after ticket creation while
  the decision engine runs in the background.

TASK: analyze_ticket_task(ticket_id, auto_assign, auto_update_priority)

  • Queue: "decision" (dedicated queue for decision engine tasks)
  • Max retries: 3 (with 30-second delay between retries)
  • Uses synchronous DB session (Celery runs in sync context)
  • Converts asyncpg URL to psycopg2 URL for sync operations

TASK FLOW:

    API Request (create ticket)
         │
         ▼
    ┌────────────────────────────────┐
    │  Return 201 Created to client  │──── Immediate response
    │  Queue analyze_ticket_task     │
    └────────────┬───────────────────┘
                 │
                 ▼ (via Redis)
    ┌────────────────────────────────┐
    │  Celery Worker picks up task   │──── Background processing
    │                                │
    │  1. Create sync DB session     │
    │  2. Fetch ticket from DB       │
    │  3. Run classify_text()        │
    │  4. Run assess_risk()          │
    │  5. Run apply_rules()          │
    │  6. Get suggestions            │
    │  7. Create DecisionLog record  │
    │  8. Auto-update priority       │
    │  9. Auto-escalate if needed    │
    │  10. Commit                    │
    └────────────────────────────────┘

ERROR HANDLING:
  • If ticket not found: return error, no retry
  • If any exception: rollback DB, retry up to 3 times with 60-second backoff

================================================================================
 18. SCORING ALGORITHMS (DETAILED)
================================================================================

═══ A. CLASSIFICATION SCORING ═══

For each category rule, the scoring function computes:

    score = 0

    FOR EACH keyword IN rule.keywords:
      IF keyword IN text (case-insensitive):
        word_count = number of words in keyword
        score += 1.0 + 0.5 × (word_count - 1)
        #  Single-word keyword = 1.0 points
        #  "locked out" (2 words) = 1.5 points
        #  "cancel subscription" (2 words) = 1.5 points

    FOR EACH pattern IN rule.patterns:
      IF regex(pattern) MATCHES text:
        score += 2.0    # Patterns are more specific → higher reward

    score *= rule.weight   # Category weight multiplier

EXAMPLE - Text: "I was charged twice for my billing subscription"

    BILLING keywords matched: "charge" (1.0), "billing" (1.0), "subscription" (1.0)
    BILLING patterns matched: none
    BILLING score = (1.0 + 1.0 + 1.0) × 1.0 = 3.0

    TECHNICAL keywords matched: none
    TECHNICAL score = 0.0

    Only BILLING scored → dominance = 1.0, absolute = 0.3
    Confidence = 0.6 × 1.0 + 0.4 × 0.3 = 0.72 → HIGH


═══ B. RISK SCORING ═══

The risk score is a sum of weighted factors:

    risk = 0.0

    # Factor 1: Category inherent risk
    risk += CATEGORY_BASE_RISK[category] × 0.3

    # Factor 2: Negative sentiment (max 0.20)
    risk += min(count_negative_words × 0.05, 0.20)

    # Factor 3: Urgency markers (max 0.15)
    risk += min(count_urgency_words × 0.05, 0.15)

    # Factor 4: Escalation triggers (max 0.15)
    risk += min(count_escalation_words × 0.05, 0.15)

    # Factor 5: Security words (max 0.15)
    risk += min(count_security_words × 0.05, 0.15)

    # Factor 6: Text length
    IF word_count < 5:   risk += 0.05
    ELIF word_count > 500: risk += 0.03

    # Factor 7: Existing priority
    IF priority = CRITICAL: risk += 0.10
    IF priority = HIGH:     risk += 0.05
    IF priority = LOW:      risk -= 0.03

    # Factor 8: Escalation flag
    IF has_escalation_flag: risk += 0.10

    # Factor 9: Low confidence amplifier
    IF confidence = LOW:    risk += 0.08
    IF confidence = MEDIUM: risk += 0.03

    risk = clamp(risk, 0.0, 1.0)

EXAMPLE - Security breach text with urgency:

    Category = SECURITY (base risk = 0.7)
    risk = 0.7 × 0.3 = 0.21                     # Factor 1
    risk += 0.05 (1 negative word)               # Factor 2: +0.05 → 0.26
    risk += 0.10 (2 urgency markers)             # Factor 3: +0.10 → 0.36
    risk += 0 (no escalation triggers)           # Factor 4: +0.00 → 0.36
    risk += 0.15 (3+ security words, capped)     # Factor 5: +0.15 → 0.51
    risk += 0 (normal text length)               # Factor 6: +0.00 → 0.51
    risk += 0 (default MEDIUM priority)          # Factor 7: +0.00 → 0.51
    risk += 0 (no escalation flag)               # Factor 8: +0.00 → 0.51
    risk += 0.03 (MEDIUM confidence)             # Factor 9: +0.03 → 0.54

    Final: risk = 0.54 → HIGH


═══ C. ROUTING SCORING ═══

    For each candidate agent with matching skill:

      workload_ratio = current_tickets / max_concurrent_tickets
      score = proficiency × (1 - workload_ratio)

    EXAMPLE:
      Agent Alice: proficiency=0.9, workload=2/10
        score = 0.9 × (1 - 0.2) = 0.9 × 0.8 = 0.72

      Agent Bob: proficiency=0.7, workload=0/10
        score = 0.7 × (1 - 0.0) = 0.7 × 1.0 = 0.70

      Agent Charlie: proficiency=0.95, workload=8/10
        score = 0.95 × (1 - 0.8) = 0.95 × 0.2 = 0.19

      Ranking: Alice (0.72) > Bob (0.70) > Charlie (0.19)
      Selected: Alice (best balance of skill and availability)

 19. ROUTING ALGORITHM (DETAILED)
================================================================================

ROUTING DECISION TREE:

    Is outcome ROUTE_AGENT, SUGGEST_RESPONSE, or ESCALATE_HUMAN?
         │
    NO ──┤──── Skip routing (no agent needed)
         │
    YES──▼
         │
    Query agents with matching skill for the intent category
         │
    ┌────▼────────────────────────────────────────────────────┐
    │  For each agent with matching skill:                    │
    │    • Check: is agent ACTIVE? Has AGENT/ADMIN role?      │
    │    • Count open + in-progress assigned tickets           │
    │    • If at max capacity → SKIP                           │
    │    • Score = proficiency × (1 - workload / max)          │
    └────┬────────────────────────────────────────────────────┘
         │
    Any skilled candidates?
         │
    YES──┤──── Sort by score DESC → return best as selected_agent
         │    + all candidates sorted
         │
    NO ──▼
         │
    ┌────▼────────────────────────────────────────────────────┐
    │  FALLBACK: Query ALL active agents (any role/skill)     │
    │    • Count each agent's current workload                │
    │    • Set skill_match_score = 0.0 (no skill match)       │
    │    • Sort by workload ASC (least loaded first)            │
    │    • Return least loaded as selected_agent               │
    └────┬────────────────────────────────────────────────────┘
         │
    Any agents at all?
         │
    YES──┤──── Return with reasoning "Fallback routing..."
         │
    NO ──▼──── Return empty (selected_agent = null, candidates = [])
