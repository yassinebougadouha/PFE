================================================================================
 1. OVERVIEW & PROJECT PURPOSE
================================================================================

The AI Support Agent Backend is a decision-centric intelligent customer support
platform. Unlike typical chatbot systems that rely solely on generative LLMs,
this platform places an explicit, rule-based decision engine at its core, using
LLMs only as linguistic interfaces for response formulation.

PROJECT GOALS:
  • Build a robust, modular, scalable backend for multi-channel customer support
  • Implement an adaptive decision engine based on confidence and risk scoring
  • Integrate Human-in-the-Loop (HITL) escalation for uncertain/high-risk cases
  • Add Visual AI capabilities for screenshot analysis and UI gap detection
  • Support RAG (Retrieval-Augmented Generation) for knowledge-grounded responses
  • Provide real-time voice agent capabilities via LiveKit
  • Ensure full traceability and auditability of every AI decision

TARGET USERS:
  1. End Customers (Clients) — Submit tickets, chat, email, WhatsApp, voice
  2. Support Agents — Handle escalated/assigned tickets, respond to customers
  3. Administrators — Configure rules, manage knowledge base, monitor metrics

WHAT MAKES THIS DIFFERENT:
  • Deterministic & Interpretable — No black-box ML for decisions.
    Every decision is traceable to matched keywords, risk factors, and rules.
  • Hybrid Architecture — Rule-based decision + LLM reformulation.
    LLMs generate responses; deterministic logic makes decisions.
  • Multi-Channel Unified — Email, Chat, WhatsApp, Voice, Tickets all feed
    into the same pipeline with channel-specific formatting.
  • Visual AI — Screenshot analysis with gap detection against reference
    screens, adaptive guidance, and UI state timeline tracking.


================================================================================
 2. DESIGN PHILOSOPHY & PRINCIPLES
================================================================================

SEPARATION OF CONCERNS:
  • Understanding (classification, scoring) is separate from Generation (LLM)
  • Decision logic is deterministic and rule-based — never delegated to LLMs
  • LLMs are used only for response reformulation and knowledge synthesis

ARCHITECTURE PRINCIPLES:
  • Backend-Heavy — All intelligence lives in the backend
  • Async-First — FastAPI async routes + Celery for background processing
  • Service Pattern — Each domain has Routes → Service → Model layers
  • Soft Deletes — Entities are never hard-deleted (is_deleted flag)
  • UUID Primary Keys — All entities use UUID v4 for distributed safety
  • Timestamp Tracking — All entities have created_at / updated_at
  • Full Audit Trail — Every significant action is logged to audit_logs

SECURITY PRINCIPLES:
  • JWT-based authentication with access + refresh tokens
  • Token blacklisting via Redis for secure logout
  • Role-based access control (CLIENT, AGENT, ADMIN)
  • Screenshot capture requires explicit user consent (opt-in)
  • Internal service-to-service auth via shared API keys

 9. CORE MODULE: SECURITY & AUTHENTICATION
================================================================================

AUTHENTICATION FLOW:

    ┌────────────────────────────────────────────────────────────┐
    │                    AUTH FLOW                                │
    └────────────────────────────────────────────────────────────┘

    1. REGISTER:
       POST /api/v1/auth/register
       → UserCreate { email, password, full_name, role }
       → Creates User with hashed password
       → Returns UserResponse

    2. LOGIN:
       POST /api/v1/auth/login
       → UserLogin { email, password }
       → Validates credentials
       → Returns TokenPair { access_token, refresh_token }
       → Access token: 15 min TTL, contains { sub: user_id, role }
       → Refresh token: 7 day TTL

    3. USE API:
       Authorization: Bearer <access_token>
       → Middleware extracts token
       → Checks Redis blacklist
       → Decodes JWT, validates type="access"
       → Loads User from DB
       → Checks user status == ACTIVE
       → Returns User object for route handlers

    4. REFRESH:
       POST /api/v1/auth/refresh
       → TokenRefreshRequest { refresh_token }
       → Validates + checks blacklist
       → Blacklists old refresh token
       → Issues new token pair

    5. LOGOUT:
       POST /api/v1/auth/logout
       → Blacklists current access token in Redis
       → Token becomes invalid until natural expiry

  ┌──────────────────┬───────────────────────────────────────────────────────┐

 16. CELERY WORKERS & ASYNC TASKS (app/workers/)
================================================================================

FILE: app/workers/celery_app.py

  Celery instance "support_worker" with:
    • Redis broker (DB 2) and result backend (DB 3)
    • JSON serialization, UTC timezone
    • Late ack + prefetch 1 (at-least-once delivery)
    • Task routing to dedicated queues (8 queues)
    • Beat schedule for periodic Gmail sync
    • Auto-discovery of tasks in: workers, decision_engine, rag, visual_ai

FILE: app/workers/tasks.py

  Sync DB session: Converts asyncpg URL to psycopg2 for Celery's sync context.

  TASK: process_email_task (queue: emails, retries: 3)
    • Converts ingested email to a ticket
    • Updates email status through lifecycle: RECEIVED → PROCESSING → CONVERTED
    • Applies auto-reply guardrails before automatic email replies
    • On failure: marks email as FAILED, retries with 60s backoff

  TASK: log_action_task (queue: logging)
    • Fire-and-forget audit logging in background
    • Creates AuditLog record from parameters

  TASK: send_notification_placeholder (queue: celery)
    • Placeholder for future notification system

  TASK: sync_gmail_for_user_task (queue: gmail, retries: 2)
    • Syncs Gmail for a specific user
    • Uses GmailSyncService with sync DB session

  TASK: send_email_reply_task (queue: gmail, retries: 2)
    • Sends reply to email via connected Gmail account
    • Uses GmailSyncService.send_reply()

  TASK: sync_all_gmail_accounts (queue: gmail, periodic)
    • Syncs all active Gmail accounts
    • Triggered by Celery Beat every GMAIL_POLL_INTERVAL_SECONDS

  TASK: process_whatsapp_incoming_task (queue: whatsapp, retries: 3)
    • Processes incoming WhatsApp message
    • Find-or-create User by phone number → Conversation → Message
    • Optionally generates and sends RAG-powered WhatsApp auto-reply
    • Uses WhatsAppSyncService

  TASK: record_whatsapp_outbound_task (queue: whatsapp)
    • Records outbound WhatsApp message in Conversation/Message

 24. MODULE: RAG KNOWLEDGE BASE (app/rag/)
================================================================================

PURPOSE:
  Retrieval-Augmented Generation system that stores, indexes, and retrieves
  knowledge articles to ground AI responses in factual company information.

EMBEDDING MODEL: all-MiniLM-L6-v2 (384 dimensions, from sentence-transformers)
VECTOR STORE: pgvector extension in PostgreSQL (cosine distance)

ARTICLE LIFECYCLE:

    ┌───────┐     ┌───────────┐     ┌──────────┐
    │ DRAFT │────▶│ PUBLISHED │────▶│ ARCHIVED │
    └───────┘     └───────────┘     └──────────┘
        │              │
        └── index ─────┘ (only published articles are searchable)

INDEXING PIPELINE:

    Article Content (text)
         │
         ▼
    ┌── RecursiveTextChunker ────────────────────────┐
    │  • Splits by: \n\n → \n → . → ? → ! → ; → , →│
    │  • Target chunk size: 512 tokens (configurable)│
    │  • Overlap: 64 tokens between chunks           │
    │  • Merges small consecutive pieces             │
    └───────────────┬────────────────────────────────┘
                    │
                    ▼
    ┌── Batch Embedding ─────────────────────────────┐
    │  • all-MiniLM-L6-v2 sentence-transformers      │
    │  • 384-dimensional vectors, L2 normalized      │
    │  • Batch size: 64                              │
    └───────────────┬────────────────────────────────┘
                    │
                    ▼
    ┌── Stored in article_chunks ────────────────────┐
    │  • Each chunk: content, embedding, token_count │
    │  • Status: PENDING → INDEXED                   │
    │  • Unique index on (article_id, chunk_index)   │
    └────────────────────────────────────────────────┘

SEARCH MODES:

  1. Semantic Search:
     • Query embedded with same model
     • pgvector cosine_distance against indexed chunks
     • Filters: PUBLISHED articles, INDEXED chunks, category, language
     • min_similarity threshold (default 0.3)
     • Returns ranked hits with similarity scores

  2. Hybrid Search:
     • Fetches 3× top_k candidates via semantic search
     • Re-ranks with weighted score:
       score = keyword_weight × kw_score + (1 - keyword_weight) × semantic
     • keyword_weight default: 0.3
     • Keyword score = fraction of query words found in chunk

PDF INGESTION:
  • PDF extraction: PyMuPDF (per-page text, metadata)
  • Creates article with source="pdf:{filename}"
  • Auto-indexes and optionally auto-publishes
  • Bulk ingestion for entire documents directory
  • Skips already-ingested PDFs (by source field)

KNOWLEDGE BASE STATS:
  • Total/published/draft/archived article counts
  • Total chunks, indexed chunks, total tokens
  • Breakdown by category and language
  • Average chunks per article

 26. MODULE: DECISION ENGINE (app/decision_engine/)
================================================================================

  (See details-decision-engine.txt for the complete 1800+ line documentation)

SUMMARY:
  The decision engine is a 9-step pipeline that automatically analyzes support
  tickets to determine how they should be handled:

  Step 1: Intent Classification — keyword + regex matching → 8 categories
  Step 2: Risk Assessment — 9 weighted factors → risk score (0.0–1.0)
  Step 3: Decision Rules — confidence × risk matrix → 5 outcomes
  Step 4: Response Suggestions — template-based per category
  Step 5: Agent Routing — skill + workload scoring
  Step 6: Escalation Package — HITL handoff for ESCALATE_HUMAN outcomes
  Step 7: Build Reasoning — human-readable explanation
  Step 8: Persist Decision Log — full audit trail
  Step 9: Auto-Actions — update priority, assign agent, escalate

DECISION OUTCOMES:
  • AUTO_RESOLVE — High confidence + low risk → system handles
  • SUGGEST_RESPONSE — Good confidence → suggest responses for agent
  • CLARIFY — Medium confidence → ask customer for more info
  • ESCALATE_HUMAN — Low confidence / high risk → human required
  • ROUTE_AGENT — No rule matched → route to skilled agent

SPECIAL RULES:
  • SECURITY category → ALWAYS escalated (priority 95)
  • CRITICAL risk → ALWAYS escalated (priority 100)
  • LOW confidence → ALWAYS escalated (priority 90)

GAP DETECTION PIPELINE (4 weighted steps):

    Observed Screenshot (analyzed)
    vs. Reference Screen (stored)
         │
         ▼
    ┌── Step 1: Visual Similarity (weight 0.40) ────────────────┐
    │  Cosine similarity of CLIP embeddings (512-dim)            │
    │  0 = identical, 1 = completely different                   │
    └───────────┬────────────────────────────────────────────────┘
                │
    ┌── Step 2: Text Diff (weight 0.30) ────────────────────────┐
    │  SequenceMatcher on OCR text (expected vs. observed)       │
    │  0 = identical, 1 = no overlap                             │
    └───────────┬────────────────────────────────────────────────┘
                │
    ┌── Step 3: Element Diff (weight 0.20) ─────────────────────┐
    │  Symmetric set difference of detected UI element types     │
    │  Reports missing and extra elements                        │
    └───────────┬────────────────────────────────────────────────┘
                │
    ┌── Step 4: Error Penalty (weight 0.10) ────────────────────┐
    │  Penalty for ERROR_MESSAGE (1.0) or LOADING_STATE (0.5)   │
    └───────────┬────────────────────────────────────────────────┘
                │
                ▼
    Gap Score → Severity:
      < 0.15: NO_GAP
      < 0.40: MINOR
      < 0.70: SIGNIFICANT
      >= 0.70: CRITICAL

UI STATE TIMELINE:
  Tracks chronological sequence of UI states per conversation:
  • Each screenshot analysis creates a UIState record
  • Auto-incremented sequence number per conversation
  • Records gap detection results (severity, details)
  • Enables temporal analysis of issue progression

ADAPTIVE GUIDANCE ENGINE:
  Rule-based + optional AI-enhanced guidance:
  • 6 rule templates: ERROR_DETECTED, WRONG_PAGE, MISSING_ELEMENT,
    LOADING_STATE, SUCCESS_STATE, TEXT_MISMATCH
  • Each template generates specific suggested actions
  • AI enhancement: calls Gemini for contextual explanation
    (enabled via VISUAL_GUIDANCE_USE_LLM = True)

VISUAL AI DATABASE TABLES:

    id, conversation_id, user_id, filename, file_path, file_size,

    id, screenshot_id, provider, ocr_text, caption, elements (JSON),
    labels, regions, embedding (Vector 512), raw_result, confidence,
    processing_ms, timestamps

  ui_states (timeline entries per conversation):
    id, conversation_id, analysis_id, screenshot_id, state_label,
    state_data, embedding, sequence_num, gap_detected, gap_details,
    gap_severity, timestamps

  reference_screens (expected UI states):
    id, name, description, screen_key (unique), file_path,
    embedding (Vector 512), expected_elements, expected_ocr_text

FULL PIPELINE (POST /visual-ai/process):
  1. Store screenshot on disk + DB
  2. Analyze with selected provider (OCR + UI + CLIP)
  3. Gap detection against reference (if provided)
  4. Add to UI state timeline
  5. Generate adaptive guidance
  6. Return: screenshot, analysis, gap_result, ui_state, guidance

 29. MODULE: WHATSAPP WEB BRIDGE (whatsapp-bridge/)
================================================================================

PURPOSE:
  Node.js HTTP bridge wrapping @whiskeysockets/baileys for free WhatsApp
  integration without an official Business account.

TECHNOLOGY: Node.js 20 + Express + Baileys (direct WhatsApp protocol)

HOW IT WORKS:
  1. On startup, connects to WhatsApp via Baileys
  2. Shows QR code at GET /qr for pairing with phone
  3. Once authenticated, persists session in /data/auth
  4. Incoming messages → forwarded to backend webhook
  5. Backend calls POST /send to send outbound messages

BRIDGE API:
  ┌────────┬────────────┬──────────────────────────────────────────────┐
  │ Method │ Path       │ Description                                   │
  ├────────┼────────────┼──────────────────────────────────────────────┤
  │ GET    │ /status    │ Connection status + phone info               │
  │ GET    │ /qr        │ QR code HTML page (auto-refresh)             │
  │ GET    │ /qr.json   │ QR code as JSON                              │
  │ POST   │ /send      │ Send message { chatId, message }             │
  │ GET    │ /health    │ Health check                                  │
  └────────┴────────────┴──────────────────────────────────────────────┘

MESSAGE FILTERING (incoming):
  • Filters out: own messages, status broadcasts, group messages,
    invalid senders, non-text messages, base64/binary content
  • Forwards to: WEBHOOK_URL (default: backend /whatsapp/bridge/webhook)
  • Payload: { id, from, body, sender_name, timestamp, type, hasMedia }

 30. MODULE: INTERNAL SERVICE-TO-SERVICE API
================================================================================

PURPOSE:
  Protected API endpoints for internal services (e.g., voice agents) to
  access RAG and AI generation without user JWT tokens.

AUTHENTICATION:
  X-Service-Key header matching INTERNAL_SERVICE_KEY from settings.
  Returns 403 if key doesn't match.

ENDPOINTS:

  POST /api/v1/internal/rag/search
    • Body: SearchRequest { query, top_k, category, language, min_similarity }
    • Returns: SearchResponse { query, hits, total_hits, model_used }
    • Used by: Voice agents (rag_bridge.py)

  POST /api/v1/internal/rag/generate
    • Body: GenerateRequest { query, channel, provider, tone, ... }
    • Returns: GenerateResponse { response, raw_response, provider, ... }
    • Used by: Voice agents for RAG-augmented answers
